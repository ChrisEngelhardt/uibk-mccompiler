3 ## Front-end
4 - Validate a given input using syntactic and semantic checks
5 - **Syntactic** checking is done by the **parser** which generates an abstract syntax tree (AST) (used for semantic checking)
6 - AST is translated to the compiler's intermediate representation (IR) and passed to the core.
7 
8 ## core
9 - Provides infrastructure for running analyses and transformations on the IR. (used for optimisation)
10 - Additional data structures, like the control flow graph (CFG), are utilised for this phase.
11 
12 ## Back-end
13 - translates the platform independent IR code to platform dependent assembly code.
14 - assembler converts this code to object code, which is finally crafted into an executable by the linker.
15 - **IR Code -> assembly code -> object code -> executable (linker)**

24 # Flex
25 
26 ## What is Flex
27 Tool that generates a scanner that divides the input into tokens. The output of flex is a C programm.
28 
29 ## Basics programm
30 Flex program consists of three sections, separated by %% lines.
31 - First section contains declarations and option settings
32 - Second section is a list of patterns and actions
33 - Third section is C code that is copied to the generated scanner
34 
35 ## Generate
36 1. flex _programmName_
37 2. gcc lex.yy.c -lfl -o _executableName_
38 3. ./_executableName_ _INPUT_ ^D
39 
40 **(Important Notes: On mac linking flex with fl seems not to work, use -ll instead)**

68 # Bison
69 
70 ## What is Bison
71 - Bisons job is to figure out the relationship among the input tokens
72 - it generates an syntax tree for further processing
73 - to define the parser, BNF will be used (slightly modified from the definition)
74 
75 ## Example
76 Simple grammer for additon and multiplication
77 
78 ```c
79 <exp> ::= <factor>
80  | <exp> + <factor>
81 <factor> ::= NUMBER
82  | <factor> * NUMBER
83 ```
84 
85 ## Basics programm
86 - Have the same three-part structure as flex
87 - The declarations here include C code to be copied to the beginning of the generated C parser, again enclosed in %{ and %}
88 - Following are %token token declarations, telling bison the names of the symbols in the parser that are tokens (convention, tokens have uppercase names)
89 - Any symbols not declared as tokens have to appear on the left side of at least one rule in the program
90 - The second section contains the rules in simplified BNF
91 - Each symbol in a bison rule has a value; the value of the target symbol
92 
93 ## Good to know
94 - Whenever a grammar has multiple levels of precedence where one kind of operator binds "tighter" than another, the parser will need a level of rule for each level
95 - It will not parse an ambiguous grammar

98 ## Example of a small calculator
99 
100 ```c
101 /* simplest version of calculator */
102 %{
103 #include <stdio.h>
104 %}
105 
106 /* declare tokens */
107 %token NUMBER
108 %token ADD SUB MUL DIV ABS
109 %token EOL
110 
111 %%
112 
113 calclist: /* nothing */ matches at beginning of input
114  | calclist exp EOL { printf("= %d\n", $2); } EOL is end of an expression
115  ;
116 
117 exp: factor default $$ = $1
118  | exp ADD factor { $$ = $1 + $3; }
119  | exp SUB factor { $$ = $1 - $3; }
120  ;
121 
122 factor: term default $$ = $1
123  | factor MUL term { $$ = $1 * $3; }
124  | factor DIV term { $$ = $1 / $3; }
125  ;
126 
127 term: NUMBER default $$ = $1
128  | ABS term { $$ = $2 >= 0? $2 : - $2; }
129 ;
130 %%
131 main(int argc, char **argv)
132 {
133  yyparse();
134 }
135 
136 yyerror(char *s)
137 {
138  fprintf(stderr, "error: %s\n", s);
139 }
140 ```
