1 # mC Compiler Specification
2 
3 This document describes the mC compiler as well as the mC language itself along with some requirements.
4 Like a regular compiler the mC compiler is divided into 3 main parts: front-end, back-end, and a core in-between.
5 
6 The front-end's task is to validate a given input using syntactic and semantic checks.
7 The syntactic checking is done by the *parser*, which, on success, generates an abstract syntax tree (AST).
8 This tree data structure is mainly used for semantic checking, although transformations can also be applied to it.
9 Moving on, the AST is translated to the compiler's intermediate representation (IR) and passed to the core.
10 Invalid inputs cause errors to be reported.
11 
12 The core provides infrastructure for running analyses and transformations on the IR.
13 These analyses and transformations are commonly used for optimisation.
14 Additional data structures, like the control flow graph (CFG), are utilised for this phase.
15 Next, the (optimised) IR is passed to the back-end.
16 
17 The back-end translates the platform *independent* IR code to platform *dependent* assembly code.
18 An assembler converts this code to *object code*, which is finally crafted into an executable by the linker.
19 For these last two steps, GCC is used — referred to as *back-end compiler* in this context.
20 
21 The mC compiler is implemented using modern C (or C++) adhering to the C11 (or C++17) standard.
22 
23 ## Milestones
24 
25 1. **Parser**
26  - Inputs are accepted / rejected correctly (syntax only).
27  - Syntactically invalid inputs result in a meaningful error message containing the corresponding source location.
28  - An AST is constructed for valid inputs.
29  - The obtained AST can be printed in the DOT format (see `mc_ast_to_dot`).
30 2. **Semantic checks**
31  - The compiler rejects semantically wrong inputs.
32  - Invalid inputs trigger a meaningful error message including source location information.
33  - Type checking can be traced (see `mc_type_check_trace`).
34  - Symbol tables can be viewed (see `mc_symbol_table`).
35 3. **Control flow graph**
36  - Valid inputs are converted to IR.
37  - The IR can be printed (see `mc_ir`).
38  - The CFG is generated and can be printed in the DOT format (see `mc_cfg_to_dot`).
39 4. **Back-end**
40  - Valid inputs are converted to IR and then to assembly code.
41  - The assembly code can be printed (see `mc_asm`).
42  - GCC is invoked to create the final executable.
43 5. **Build Infrastructure**
44  - Your code builds and tests successfully on my evaluation system.

46 ## mC Language
47 
48 This section defines *mC* — a simple, C-like language.
49 The semantics of mC are identical to C unless specified otherwise.
50 
51 ### Grammar
52 
53 The next segment defines the grammar of mC using this notation:
54 
55 - `#` starts a single line comment
56 - `,` indicates concatenation
57 - `|` indicates alternation
58 - `( )` indicates grouping
59 - `[ ]` indicates optional parts (0 or 1)
60 - `{ }` indicates repetition (1 or more)
61 - `[ ]` and `{ }` can be combined to build *0 or more* repetition
62 - `" "` indicates a terminal string
63 - `/ /` indicates a [RegEx](https://www.regular-expressions.info/)
64 
65 ```
66 # Primitives
67 
68 alpha = /[a-zA-Z_]/
69 
70 alpha_num = /[a-zA-Z0-9_]/
71 
72 digit = /[0-9]/
73 
74 identifier = alpha , [ { alpha_num } ]
75 
76 
77 # Operators
78 
79 unary_op = "-" | "!"
80 
81 binary_op = "+" | "-" | "*" | "/"
82  | "<" | ">" | "<=" | ">="
83  | "&&" | "||"
84  | "==" | "!="
85 
86 
87 # Types
88 
89 type = "bool" | "int" | "float" | "string"
90 
91 
92 # Literals
93 
94 literal = bool_literal
95  | int_literal
96  | float_literal
97  | string_literal
98 
99 bool_literal = "true" | "false"
100 
101 int_literal = { digit }
102 
103 float_literal = { digit } , "." , { digit }
104 
105 string_literal = /"[^"]*"/
106 
107 
108 # Declarations / Assignments
109 
110 declaration = type , [ "[" , int_literal , "]" ] , identifier
111 
112 assignment = identifier , [ "[" , expression , "]" ] , "=" , expression
113 
114 
115 # Expressions
116 
117 expression = literal
118  | identifier , [ "[" , expression , "]" ]
119  | call_expr
120  | unary_op , expression
121  | expression , binary_op , expression
122  | "(" , expression , ")"
123 
124 
125 # Statements
126 
127 statement = if_stmt
128  | while_stmt
129  | ret_stmt
130  | declaration , ";"
131  | assignment , ";"
132  | expression , ";"
133  | compound_stmt
134 
135 if_stmt = "if" , "(" , expression , ")" , statement , [ "else" , statement ]
136 
137 while_stmt = "while" , "(" , expression , ")" , statement
138 
139 ret_stmt = "return" , [ expression ] , ";"
140 
141 compound_stmt = "{" , [ { statement } ] , "}"
142 
143 
144 # Function Definitions / Calls
145 
146 function_def = ( "void" | type ) , identifier , "(" , [ parameters ] , ")" , compound_stmt
147 
148 parameters = declaration , [ { "," , declaration } ]
149 
150 call_expr = identifier , "(" , [ arguments ] , ")"
151 
152 arguments = expression , [ { "," expression } ]
153 
154 
155 # Program
156 
157 program = [ { function_def } ]
158 ```

160 ### Comments
161 
162 mC supports only *C-style* comments, starting with `/*` and ending with `*/`.
163 Like in C, they can span across multiple lines.
164 Comments are discarded by the parser; however, line breaks are taken into account for line numbering.
165 
166 ### Size Limitations
167 
168 `long` and `double` are used in the compiler to store mC's `int` and `float` literals, respectively.
169 It is assumed that both types are big and precise enough to store the corresponding literal.
170 
171 Furthermore, it is assumed that arrays and strings are at most `LONG_MAX` elements long.
172 
173 ### Special Semantics
174 
175 #### Boolean
176 
177 `bool` is considered a first-class citizen, distinct from `int`.
178 
179 The operators `!`, `&&`, and `||` can only be used with Booleans.
180 Short-circuit evaluation is *not* supported.
181 
182 #### Strings
183 
184 Strings are immutable and do not support any operation (e.g. concatenation).
185 Like comments, strings can span across multiple lines.
186 Newlines and indentation whitespaces are part of the string, when dealing with multiline strings.
187 Escape sequences are *not* supported.
188 
189 Their sole purpose is to be used with the built-in `print` function (see below).
190 
191 #### Arrays
192 
193 Only one dimensional arrays with static size are supported.
194 The size must be stated during declaration and is part of the type.
195 The following statement declares an array of integers with 42 elements.
196 
197  int[42] my_array;
198 
199 We do not support *any* operations on whole arrays.
200 For example, the following code is *invalid*:
201 
202  int[10] a;
203  int[10] b;
204  int[10] c;
205 
206  c = a + b; /* not supported */
207 
208 This needs to be rewritten as a loop in order to work:
209 
210  int i;
211  i = 0;
212  while (i < 10) {
213  c[i] = a[i] + b[i];
214  i = i + 1;
215  }
216 
217 Even further, one cannot assign to a variable of array type.
218 
219  c = a; /* not supported, even though both are of type int[10] */
220 
221 #### Call by Value / Call by Reference
222 
223 `bool`, `int`, and `float` are passed directly (by value).
224 Strings and arrays are passed via pointers internally (by reference).
225 
226 Modifications made to an array inside a function are visible outside of that function.
227 
228 #### Type Conversion
229 
230 There are *no* type conversion, neither implicit nor explicit.
231 
232 An expression used as a condition (for `if` or `while`) is expected to be of type `bool`.
233 
234 *Note:* If the need for explicit type conversion arises, additional built-ins will be added for this purpose.
235 
236 #### Entry Point
237 
238 The top-level grammar rule is `program` which simply consists of 0 or more function definitions.
239 
240 While the parser happily accepts empty source files, a semantic check enforces the presence of a function named `main`.
241 This function takes no arguments and returns an `int`.
242 
243 On success, an mC program returns `0`.
244 
245 #### Declaration, Definition, and Initialization
246 
247 `declaration` is used to declare variables which can then be initialised with `assignment`.
248 Splitting declaration and initialisation simplifies the creation of symbol tables.
249 
250 Functions are always declared by their definition.
251 Forward declarations are therefore *not* supported.
252 Contrary to C, it is possible to call a function before it has been declared (in case of mC defined).
253 
254 #### Empty Parameter List
255 
256 In C, the parameter list of a function taking no arguments is written as `(void)`.
257 mC, in this case, just uses an empty parameter list `()`.
258 
259 #### Dangling Else
260 
261 A [*dangling else*](https://en.wikipedia.org/wiki/Dangling_else) belongs to the innermost `if`.
262 The following mC code snippets are semantically equivalent:
263 
264  if (c1) | if (c1) {
265  if (c2) | if (c2) {
266  f2(); | f2();
267  else | } else {
268  f3(); | f3();
269  | }
270  | }
271 
272 ### Built-in Functions
273 
274 The following built-in functions are provided by the compiler for I/O operations:
275 
276 - `void print(string)` outputs the given string to `stdout`
277 - `void print_nl()` outputs the new-line character (`\n`) to `stdout`
278 - `void print_int(int)` outputs the given integer to `stdout`
279 - `void print_float(float)` outputs the given float to `stdout`
280 - `int read_int()` reads an integer from `stdin`
281 - `float read_float()` reads a float from `stdin`

283 ## mC Compiler
284 
285 The mC compiler is implemented as a library.
286 It can be used either programmatically or via the provided command-line applications.
287 
288 The focus lies on a clean and modular implementation as well as a straight forward architecture, rather than raw performance.
289 For example, each semantic check may traverse the AST in isolation.
290 
291 The compiler guarantees the following:
292 
293 - Exported symbols are prefixed with `mcc_`.
294 - It is thread-safe.
295 - No memory is leaked — even in error cases.
296 - Functions do not interact directly with `stdin`, `stdout`, or `stderr`.
297 - No function terminates the application on correct usage.
298 
299 ### Logging
300 
301 Logging infrastructure may be present; however, all log output is disabled by default.
302 The log level can be set with the environment variable `MCC_LOG_LEVEL`.
303 
304  0 = no logging
305  1 = normal logging (info)
306  2 = verbose logging (debug)
307 
308 The output destination can be set with `MCC_LOG_FILE` and defaults to `stdout`.
309 
310 Log messages do not overlap on multi-threaded execution.
311 
312 ### Parser
313 
314 The parser reads the given input and, if it conforms syntactically to an mC program, constructs the corresponding AST.
315 An invalid input is rejected, resulting in a meaningful error message.
316 For instance:
317 
318  foo.mc:3:8: error: unexpected '{', expected ‘(’
319 
320 It is recommended to closely follow the error message format of other compilers.
321 This allows for better IDE integration.
322 
323 Displaying the offending source code along with the error message is helpful, but not required.
324 
325 Parsing may stop on the first error.
326 Error recovery is optional.
327 
328 The parser component may be generated by tools like `flex` and `bison`, or similar.
329 Although, you are encouraged to implement a recursive descent or combinator parser instead.
330 Nevertheless, pay attention to operator precedence.
331 
332 Note that partial mC programs, like an expression or statement, are not valid inputs to the main *parse* function.
333 The library may provide additional functions for parsing single expressions or statements.
334 
335 ### Abstract Syntax Tree
336 
337 The AST data structure itself is *not* specified.
338 
339 Consider using the visitor pattern for tree traversals.
340 
341 Given this example input:
342 
343 ```c
344 int fib(int n)
345 {
346  if (n < 2) return n;
347  return fib(n - 1) + fib(n - 2);
348 }
349 ```
350

355 ### Semantic Checks
356 
357 As the parser only does syntactic checking, additional semantic checks are implemented:
358 
359 - Checking for uses of undeclared variables
360 - Checking for multiple declarations of variables with the same name in the same scope
361 - Checking for multiple definitions of functions with the same name
362 - Checking for calls to unknown functions
363 - Checking for presence of `main` and correct signature
364 - Checking that all execution paths of a non-void function return a value
365 - Type checking (remember, neither implicit nor explicit type conversions)
366  - Includes checking operations on arrays
367  - Includes checking arguments and return types for call expressions
368 
369 In addition to the AST, *symbol tables* are created and used for semantic checking.
370 Be sure to correctly model [*shadowing*](https://en.wikipedia.org/wiki/Variable_shadowing).
371 
372 ### Intermediate Representation
373 
374 As IR, a low-level [three-address code (TAC)](https://en.wikipedia.org/wiki/Three-address_code) is used.
375 The instruction set of this code is *not* specified.
376 
377 The compiler's core is independent from the front- and back-end.
378 
379 ### Control Flow Graph
380 
381 A control flow graph data structure is present and can be constructed for a given IR program.
382 This graph is commonly used by analyses for extracting structural information crucial for transformation steps.
383 
384 It is recommended to also provide a visitor mechanism for this graph.
385 
386 Like the AST, it can be visualised.
387 The example below is taken from [Marc Moreno Maza](http://www.csd.uwo.ca/~moreno/CS447/Lectures/CodeOptimization.html/node6.html).
388 
389 Given this example IR:
390 
391 ```
392  s = 0
393  i = 0
394  n = 10
395 L1: t1 = a - b
396  ifz t1 goto L2
397  t2 = i * 4
398  s = s + t2
399  goto L3
400 L2: s = s + i
401 L3: i = i + 1
402  t3 = n - i
403  ifnz t3 goto L1
404  t4 = a - b
405 ```
406 
407 The visualisation of the corresponding CFG could look like this:
408 
409 ![CFG Example](images/cfg.png)

411 ### Assembly Code Generation
412 
413 The mC compiler targets x86 and uses GCC as back-end compiler.
414 On an x86_64 system, GCC multi-lib support must be available and the flag `-m32` is passed to the compiler.
415 
416 The code generated by the back-end is compiled with the [GNU Assembler](https://en.wikipedia.org/wiki/GNU_Assembler) (by GCC).
417 Pay special attention to floating point and integer handling.
418 
419 Use [cdecl calling convention](https://en.wikipedia.org/wiki/X86_calling_conventions#cdecl).
420 It is paramount to correctly implement the calling convention, otherwise the stack may get corrupted during function calls and returns.

422 ## Applications
423 
424 Apart from the main compiler executable `mcc`, additional auxiliary executables are provided.
425 These executables aid the development process and are used for evaluation.
426 
427 The applications are commonly defined by their usage information.
428 Composing them with other command-line tools, like `dot`, is a core feature.
429 
430 The exact output format is not specified in all cases.
431 However, details should *not* be omitted — like simplifying the AST.
432 
433 All applications exit with code `EXIT_SUCCESS` *iff* they succeeded in their operation.
434 
435 Each executable accepts multiple input files.
436 The inputs are parsed in isolation; the resulting ASTs are merged before semantic checks are run.
437

639 ## Coding Guidelines
640 
641 Architectural design and readability of your code will be judged.
642 
643 - Don't be a git — use [Git](https://git-scm.com/)!
644 - Files are UTF-8 encoded and use Unix line-endings (`\n`).
645 - Files contain *one* newline at the end.
646 - Lines do not contain trailing whitespaces.
647 - Your code does not trigger warnings, justify them if otherwise.
648 - Do not waste time or space (this includes memory leaks).
649 - Check for leaks using `valgrind`, especially in error cases.
650 - Keep design and development principles in mind, especially KISS and DRY.
651 - Always state the source of non-original content.
652  - Use persistent links when possible.
653  - Ideas and inspirations should be referenced too.
654 
655 > Credit where credit is due.
656 
657 ### C/C++
658 
659 - While not required, it is highly recommended to use a formatting tool, like [ClangFormat](https://clang.llvm.org/docs/ClangFormat.html).
660  A configuration file is provided in the getting started code-base, however, you are free to rule your own.
661 - Lines should not exceed 120 columns.
662 - The nesting depth of control statements should not exceed 4.
663  - Move inner code to dedicated functions or macros.
664  - Avoid using conditional and loop statements inside `case`.
665 - Use comments *where necessary*.
666  - Code should be readable and tell *what* is happening.
667  - A comment should tell you *why* something is happening, or what to look out for.
668  - An overview at the beginning of a module header is welcome.
669 - Use the following order for includes (separated by an empty line):
670  - Corresponding header (`ast.c` → `ast.h`)
671  - System headers
672  - Other library headers
673  - Public headers of the same project
674  - Private headers of the same project
675 - The structure of a source file should be similar to its corresponding header file.
676  - Separators can be helpful, but they should not distract the reader.
677 - Keep public header files free from implementation details, this also applies to the overview comment.
678 - Use assertions to verify preconditions.
679 - Ensure the correct usage of library functions, remember to always check return codes.
680 - Prefer bound-checking functions, like `snprintf`, over their non-bound-checking variant.
681 
682 Also, keep the following in mind, taken from [Linux Kernel Coding Style](https://www.kernel.org/doc/html/v4.10/process/coding-style.html):
683 
684 > Functions should be short and sweet, and do just one thing.
685 > They should fit on one or two screenfuls of text (the ISO/ANSI screen size is 80x24, as we all know), and do one thing and do that well.
686 >
687 > The maximum length of a function is inversely proportional to the complexity and indentation level of that function.
688 > So, if you have a conceptually simple function that is just one long (but simple) case-statement, where you have to do lots of small things for a lot of different cases, it's OK to have a longer function.
