1 <h2>Chapter&nbsp;1.&nbsp;Introducing Flex and Bison</h2><p>Flex and Bison are tools for building programs that handle
2  structured input. They were originally tools for building compilers, but
3  they have proven to be useful in many other areas. In this first chapter,
4  we’ll start by looking at a little (but not too much) of the theory behind
5  them, and then we’ll dive into some examples of their use.</p><div><h2>Lexical Analysis and Parsing</h2><p>The earliest compilers back in the 1950s used utterly ad
6  hoc techniques to analyze the syntax of the source code of programs they
7  were compiling. During the 1960s, the field got a lot of academic
8  attention, and by the early 1970s, syntax analysis was a well-understood
9  field.</p><p>One of the key insights was to break the job into two parts:
10  <span><em>lexical analysis</em></span> (also called
11  <span><em>lexing</em></span> or <span><em>scanning</em></span>) and
12  <span><em>syntax analysis</em></span> (or
13  <span><em>parsing</em></span>).</p><p>Roughly speaking, scanning divides the input into
14  meaningful chunks, called <span><em>tokens</em></span>, and parsing
15  figures out how the tokens relate to each other. For example, consider
16  this snippet of C code:</p><div style="overflow: auto;"><pre>alpha = beta + gamma ;</pre></div><p>A scanner divides this into the tokens <code>alpha</code>, <code>equal sign</code>, <code>beta</code>, <code>plus sign</code>, <code>gamma</code>, and <code>semicolon</code>. Then the parser determines that
17  <code>beta + gamma</code> is an expression, and
18  that the expression is assigned to <code>alpha</code>.</p><div><div>Getting Flex and Bison</div><p>Most Linux and BSD systems come with flex and bison as
19  part of the base system. If your system doesn’t have them, or has
20  out-of-date versions, they’re both easy to install.</p><p>Flex is a Sourceforge project, at <a href="http://flex.sourceforge.net/">http://flex.sourceforge.net/</a>. The current version as of
21  early 2009 was 2.5.35. Changes from version to version are usually
22  minor, so it’s not essential to update your version if it’s close to
23  .35, but some systems still ship with version 2.5.4 or 2.5.4a, which
24  is more than a decade old.</p><p>Bison is available from <a href="http://www.gnu.org/software/bison/">http://www.gnu.org/software/bison/</a>. The current version
25  as of early 2009 was 2.4.1. Bison is under fairly active development,
26  so it’s worth getting an up-to-date version to see what’s new. Version
27  2.4 added support for parsers in Java, for example. BSD users can
28  generally install a current version of flex or bison using the ports
29  collection. Linux users may be able to find current RPMs. If not, flex
30  and bison both use the standard GNU build process, so to install them,
31  download and unpack the current flex and bison tarballs from the web
32  sites, run <code>./configure</code> and then
33  <code>make</code> to build each, then become
34  superuser and <code>make install</code> to
35  install them.</p><p>Flex and bison both depend on the GNU m4 macroprocessor. Linux
36  and BSD should all have m4, but in case they don’t, or they have an
37  ancient version, the current GNU m4 is at <a href="http://www.gnu.org/software/m4/">http://www.gnu.org/software/m4/</a>.</p><p>For Windows users, both bison and flex are included in the
38  Cygwin Linux emulation environment available at <a href="http://www.cygwin.com/">http://www.cygwin.com/</a>. You can use the C or C++ code
39  they generate either with the Cygwin development tools or with native
40  Windows development tools.</p></div></div><div><h2>Regular Expressions and Scanning</h2><p>Scanners generally work by looking for patterns of
41  characters in the input. For example, in a C program, an integer
42  constant is a string of one or more digits, a variable name is a letter
43  followed by zero or more letters or digits, and the various operators
44  are single characters or pairs of characters. A straightforward way to
45  describe these patterns is <span><em>regular expressions</em></span>,
46  often shortened to <span><em>regex</em></span> or
47  <span><em>regexp</em></span>. These are the same kind of patterns that
48  the editors ed and vi and the search program egrep use to describe text
49  to search for. A flex program basically consists of a list of regexps
50  with instructions about what to do when the input matches any of them,
51  known as <span><em>actions</em></span>. A flex-generated scanner reads through its input,
52  matching the input against all of the regexps and doing the appropriate
53  action on each match. Flex translates all of the regexps into an
54  efficient internal form that lets it match the input against all the
55  patterns simultaneously, so it’s just as fast for 100 patterns as for
56  one.<sup>[<a href="https://www.oreilly.com/library/view/flex-bison/9780596805418/ch01.html#ftn.id333485">1</a>]</sup></p><div><h3>Our First Flex Program</h3><p>Unix systems (by which I also mean Unix-ish systems
57  including Linux and the BSDs) come with a word count program, which
58  reads through a file and reports the number of lines, words, and
59  characters in the file. Flex lets us write <code>wc</code> in a few dozen lines, shown in <a href="https://www.oreilly.com/library/view/flex-bison/9780596805418/ch01.html#pgmwc">Example&nbsp;1-1</a>.</p><div><p>Example&nbsp;1-1.&nbsp;Word count fb1-1.l</p><div style="overflow: auto;"><pre>/* just like Unix wc */
60 %{
61 int chars = 0;
62 int words = 0;
63 int lines = 0;
64 %}
65 
66 %%
67 
68 [a-zA-Z]+ { words++; chars += strlen(yytext); }
69 \n { chars++; lines++; }
70 . { chars++; }
71 
72 %%
73 
74 main(int argc, char **argv)
75 {
76  yylex();
77  printf("%8d%8d%8d\n", lines, words, chars);
78 }</pre></div></div><p>Much of this program should look familiar to C programmers,
79  since most of it is C. A flex program consists of three sections,
80  separated by <code>%%</code> lines. The first
81  section contains declarations and option settings. The second section
82  is a list of patterns and actions, and the third section is C code
83  that is copied to the generated scanner, usually small routines
84  related to the code in the actions.</p><p>In the declaration section, code inside of <code>%{</code> and <code>%}</code>
85  is copied through verbatim near the beginning of the generated C
86  source file. In this case it just sets up variables for lines, words,
87  and characters.</p><p>In the second section, each pattern is at the beginning of a
88  line, followed by the C code to execute when the pattern matches. The
89  C code can be one statement or possibly a multiline block in braces,
90  <code>{ }</code>. (Each pattern
91  <span><em>must</em></span> start at the beginning of the line, since
92  flex considers any line that starts with whitespace to be code to be
93  copied into the generated C program.)</p><p>In this program, there are only three patterns. The first one,
94  <code>[a-zA-Z]+</code>, matches a word. The
95  characters in brackets, known as a <span><em>character
96  class</em></span>, match any single upper- or lowercase letter, and the
97  <code>+</code> sign means to match one or more
98  of the preceding thing, which here means a string of letters or a
99  word. The action code updates the number of words and characters seen.
100  In any flex action, the variable <code>yytext</code> is set to point to the input text
101  that the pattern just matched. In this case, all we care about is how
102  many characters it was so we can update the character count
103  appropriately.</p><p>The second pattern, <code>\n</code>, just
104  matches a new line. The action updates the number of lines and
105  characters.</p><p>The final pattern is a dot, which is regex-ese for any
106  character. (It’s similar to a <code>?</code> in
107  shell scripts.) The action updates the number of characters. And
108  that’s all the patterns we need.<sup>[<a href="https://www.oreilly.com/library/view/flex-bison/9780596805418/ch01.html#ftn.id347971">2</a>]</sup></p><p>The C code at the end is a main program that calls <code>yylex()</code>, the name that flex gives to the
109  scanner routine, and then prints the results. In the absence of any
110  other arrangements, the scanner reads from the standard input. So
111  let’s run it.</p><div style="overflow: auto;"><pre>$ flex fb1-1.l
112 $ cc lex.yy.c -lfl
113 $ ./a.out
114 The boy stood on the burning deck
115 shelling peanuts by the peck
116 ^D
117 2 12 63
118 $</pre></div><p>First we tell flex to translate our program, and in classic Unix
119  fashion since there are no errors, it does so and says nothing. Then
120  we compile <code>lex.yy.c</code>, the C program
121  it generated; link it with the flex library, <code>-lfl</code>; run it; and type a little input for it
122  to count. Seems to work.</p><p>The actual <code>wc</code> program uses a
123  slightly different definition of a word, a string of non-whitespace
124  characters. Once we look up what all the whitespace characters are, we
125  need only replace the line that matches words with one that matches a
126  string of non-whitespace characters:</p><div style="overflow: auto;"><pre>[^ \t\n\r\f\v]+ { words++; chars += strlen(yytext); }</pre></div><p>The <code>^</code> at the beginning of the
127  character class means to match any character other than the ones in
128  the class, and the <code>+</code> once again
129  means to match one or more of the preceding patterns. This
130  demonstrates one of flex’s strengths—it’s easy to make small changes
131  to patterns and let flex worry about how they might affect the
132  generated code.</p></div><div><h3>Programs in Plain Flex</h3><p>Some applications are simple enough that you can write the whole
133  thing in flex, or in flex with a little bit of C. For example, <a href="https://www.oreilly.com/library/view/flex-bison/9780596805418/ch01.html#etoalex">Example&nbsp;1-2</a> shows the skeleton of a translator from English
134  to American.</p><div><p>Example&nbsp;1-2.&nbsp;English to American fb1-2.l</p><div style="overflow: auto;"><pre>/* English -&gt; American */
135 %%
136 "colour" { printf("color"); }
137 "flavour" { printf("flavor"); }
138 "clever" { printf("smart"); }
139 "smart" { printf("elegant"); }
140 "conservative" { printf("liberal"); }
141  … lots of other words …
142 . { printf("%s", yytext); }
143 %%</pre></div></div><p>It reads through its input, printing the American version when
144  it matches an English word and passing everything else through. This
145  example is somewhat unrealistic (<span><em>smart</em></span> can also
146  mean hurt, after all), but flex is not a bad tool to use for doing
147  modest text transformations and for programs that collect statistics
148  on input. More often than not, though, you’ll want to use flex to
149  generate a scanner that divides the input into tokens that are then
150  used by other parts of your program.</p></div><div><h3>Putting Flex and Bison Together</h3><p>The first program we’ll write using both flex and bison
151  is a desk calculator. First we’ll write a scanner, and then we’ll
152  write a parser and splice the two of them together.</p><p>To keep things simple, we’ll start by recognizing only integers,
153  four basic arithmetic operators, and a unary absolute value operator
154  (<a href="https://www.oreilly.com/library/view/flex-bison/9780596805418/ch01.html#calclex1">Example&nbsp;1-3</a>).</p><div><p>Example&nbsp;1-3.&nbsp;A simple flex scanner fb1-3.l</p><div style="overflow: auto;"><pre>/* recognize tokens for the calculator and print them out */
155 %%
156 "+" { printf("PLUS\n"); }
157 "-" { printf("MINUS\n"); }
158 "*" { printf("TIMES\n"); }
159 "/" { printf("DIVIDE\n"); }
160 "|" { printf("ABS\n"); }
161 [0-9]+ { printf("NUMBER %s\n", yytext); }
162 \n { printf("NEWLINE\n"); }
163 [ \t] { }
164 . { printf("Mystery character %s\n", yytext); }
165 %%</pre></div></div><p>The first five patterns are literal operators, written as quoted
166  strings, and the actions, for now, just print a message saying what
167  matched. The quotes tell flex to use the strings as is, rather than
168  interpreting them as regular expressions.</p><p>The sixth pattern matches an integer. The bracketed pattern
169  <code>[0-9]</code>matches any single digit, and
170  the following <code>+</code> sign means to match
171  one or more of the preceding item, which here means a string of one or
172  more digits. The action prints out the string that’s matched, using
173  the pointer <code>yytext</code> that the scanner
174  sets after each match.</p><p>The seventh pattern matches a newline character, represented by
175  the usual C <code>\n</code> sequence.</p><p>The eighth pattern ignores whitespace. It matches any single
176  space or tab (<code>\t</code>), and the empty
177  action code does nothing.</p><p>The final pattern is the catchall to match anything the other
178  patterns didn’t. Its action code prints a suitable complaint.</p><p>These nine patterns now provide rules to match anything that the
179  user might enter. As we continue to develop the calculator, we’ll add
180  more rules to match more tokens, but these will do to get us
181  started.</p><p>In this simple flex program, there’s no C code in the third
182  section. The flex library (<code>-lfl</code>) provides a tiny main program
183  that just calls the scanner, which is adequate for this
184  example.</p><p>So let’s try out our scanner:</p><div style="overflow: auto;"><pre>$ flex fb1-3.l
185 $ cc lex.yy.c -lfl
186 $ ./a.out

190 NUMBER 34

192  5 6 / 7q
193 NUMBER 5
194 NUMBER 6
195 DIVIDE
196 NUMBER 7
197 Mystery character q
198 NEWLINE
199 ^D
200 $</pre></div><p>First we run flex, which translates the scanner into a C program
201  called <code>lex.yy.c</code>, then we compile
202  the C program, and finally we run it. The output shows that it
203  recognizes numbers as numbers, it recognizes operators as operators,
204  and the <code>q</code> in the last line of input
205  is caught by the catchall pattern at the end. (That <code>^D</code> is a Unix/Linux end-of-file character. On
206  Windows you’d type <code>^Z</code>.)</p></div><div><h3>The Scanner as Coroutine</h3><p>Most programs with flex scanners use the scanner to
207  return a stream of tokens that are handled by a parser. Each time the
208  program needs a token, it calls <code>yylex()</code>, which reads a little input and
209  returns the token. When it needs another token, it calls <code>yylex()</code> again. The scanner acts as a
210  coroutine; that is, each time it returns, it remembers where it was,
211  and on the next call it picks up where it left off.</p><p>Within the scanner, when the action code has a token ready, it
212  just returns it as the value from <code>yylex()</code>. The next time the program calls
213  <code>yylex()</code>, it resumes scanning with
214  the next input characters. Conversely, if a pattern doesn’t produce a
215  token for the calling program and doesn’t return, the scanner will
216  just keep going within the same call to <code>yylex()</code>, scanning the next input characters.
217  This incomplete snippet shows two patterns that return tokens, one for
218  the <code>+</code> operator and one for a
219  number, and a whitespace pattern that does nothing, thereby ignoring
220  what it matched.</p><div style="overflow: auto;"><pre>"+" { return ADD; }
221 [0-9]+ { return NUMBER; }
222 [ \t] { /* ignore whitespace */ }</pre></div><p>This apparent casualness about whether action code returns often
223  confuses new flex users, but the rule is actually quite simple: If
224  action code returns, scanning resumes on the next call to <code>yylex()</code>; if it doesn’t return, scanning
225  resumes immediately.</p><p>Now we’ll modify our scanner so it returns tokens that a parser
226  can use to implement a calculator.</p></div><div><h3>Tokens and Values</h3><p>When a flex scanner returns a stream of tokens, each
227  token actually has two parts, the token and the token’s
228  <span><em>value</em></span>. The token is a small integer. The token
229  numbers are arbitrary, except that token zero always means
230  end-of-file. When bison creates a parser, bison assigns the token
231  numbers automatically starting at 258 (this avoids collisions with
232  literal character tokens, discussed later) and creates a <code>.h</code> with definitions of the tokens numbers.
233  But for now, we’ll just define a few tokens by hand:</p><div style="overflow: auto;"><pre>NUMBER = 258,
234 ADD = 259,
235 SUB = 260,
236 MUL = 261,
237 DIV = 262,
238 ABS = 263,
239 EOL = 264 end of line</pre></div><p>(Well, actually, it’s the list of token numbers that bison will
240  create, as we’ll see a few pages ahead. But these token numbers are as
241  good as any.)</p><p>A token’s value identifies which of a group of similar tokens
242  this one is. In our scanner, all numbers are <code>NUMBER</code> tokens, with the value saying what
243  number it is. When parsing more complex input with names,
244  floating-point numbers, string literals, and the like, the value says
245  which name, number, literal, or whatever, this token is. Our first
246  version of the calculator’s scanner, with a small main program for
247  debugging, is in <a href="https://www.oreilly.com/library/view/flex-bison/9780596805418/ch01.html#clex1">Example&nbsp;1-4</a>.</p><div><p>Example&nbsp;1-4.&nbsp;Calculator scanner fb1-4.l</p><div style="overflow: auto;"><pre>/* recognize tokens for the calculator and print them out */
248 %{
249  enum yytokentype {
250  NUMBER = 258,
251  ADD = 259,
252  SUB = 260,
253  MUL = 261,
254  DIV = 262,
255  ABS = 263,
256  EOL = 264
257  };
258 
259  int yylval;
260 %}
261 
262 %%
263 "+" { return ADD; }
264 "-" { return SUB; }
265 "*" { return MUL; }
266 "/" { return DIV; }
267 "|" { return ABS; }
268 [0-9]+ { yylval = atoi(yytext); return NUMBER; }
269 \n { return EOL; }
270 [ \t] { /* ignore whitespace */ }
271 . { printf("Mystery character %c\n", *yytext); }
272 %%
273 main(int argc, char **argv)
274 {
275  int tok;
276 
277  while(tok = yylex()) {
278  printf("%d", tok);
279  if(tok == NUMBER) printf(" = %d\n", yylval);
280  else printf("\n");
281  }
282 }</pre></div></div><p>We define the token numbers in a C <code>enum</code>. Then we make <code>yylval</code>, the variable that stores the token
283  value, an integer, which is adequate for the first version of our
284  calculator. (Later we’ll see that the value is usually defined as a
285  union so that different kinds of tokens can have different kinds of
286  values, e.g., a floating-point number or a pointer to a symbol’s entry
287  in a symbol table.)</p><p>The list of patterns is the same as in the previous example, but
288  the action code is different. For each of the tokens, the scanner
289  returns the appropriate code for the token; for numbers, it turns the
290  string of digits into an integer and stores it in <code>yylval</code> before returning. The pattern that
291  matches whitespace doesn’t return, so the scanner just continues to
292  look for what comes next.</p><p>For testing only, a small main program calls <code>yylex()</code>, prints out the token values, and,
293  for <code>NUMBER</code> tokens, also prints
294  <code>yylval</code>.</p><div style="overflow: auto;"><pre>$ flex fb1-4.l
295 $ cc lex.yy.c -lfl
296 $ ./a.out
297 a / 34 + |45
298 Mystery character a
299 262
300 258 = 34
301 259
302 263
303 258 = 45
304 264
305 ^D
306 $</pre></div><p>Now that we have a working scanner, we turn our attention to
307  parsing.</p><div><p>Where Did Flex and Bison Come From?</p><p>Bison is descended from yacc, a parser generator written
308  between 1975 and 1978 by Stephen C. Johnson at Bell Labs. As its
309  name, short for “yet another compiler compiler,” suggests, many
310  people were writing parser generators at the time. Johnson’s tool
311  combined a firm theoretical foundation from parsing work by D. E.
312  Knuth, which made its parsers extremely reliable, and a convenient
313  input syntax. These made it extremely popular among users of Unix
314  systems, although the restrictive license under which Unix was
315  distributed at the time limited its use outside of academia and the
316  Bell System. In about 1985, Bob Corbett, a graduate student at the
317  University of California, Berkeley, reimplemented yacc using
318  somewhat improved internal algorithms, which evolved into Berkeley
319  yacc. Since his version was faster than Bell’s yacc and was
320  distributed under the flexible Berkeley license, it quickly became
321  the most popular version of yacc. Richard Stallman of the Free Software
322  Foundation (FSF) adapted Corbett’s work for use in the GNU project,
323  where it has grown to include a vast number of new features as it
324  has evolved into the current version of bison. Bison is now
325  maintained as a project of the FSF and is distributed under the GNU
326  Public License.</p><p>In 1975, Mike Lesk and summer intern Eric Schmidt wrote lex, a
327  lexical analyzer generator, with most of the programming being done
328  by Schmidt. They saw it both as a standalone tool and as a companion
329  to Johnson’s yacc. Lex also became quite popular, despite being
330  relatively slow and buggy. (Schmidt nonetheless went on to have a
331  fairly successful career in the computer industry where he is now
332  the CEO of Google.)</p><p>In about 1987, Vern Paxson of the Lawrence Berkeley Lab took a
333  version of lex written in ratfor (an extended Fortran popular at the
334  time) and translated it into C, calling it flex, for
335  “<span><em>F</em></span>ast <span><em>Lex</em></span>ical Analyzer
336  Generator.” Since it was faster and more reliable than AT&amp;T lex
337  and, like Berkeley yacc, available under the Berkeley license, it
338  has completely supplanted the original lex. Flex is now a
339  SourceForge project, still under the Berkeley license.</p></div></div></div><div><h2>Grammars and Parsing</h2><p>The parser’s job is to figure out the relationship among
340  the input tokens. A common way to display such relationships is a
341  <span><em>parse tree</em></span>. For example, under the usual rules of
342  arithmetic, the arithmetic expression <code>1 * 2 + 3 * 4 + 5</code> would have the parse tree in <a href="https://www.oreilly.com/library/view/flex-bison/9780596805418/ch01.html#ptree1">Figure&nbsp;1-1</a>.</p><div><div><img src="https://www.oreilly.com/library/view/flex-bison/9780596805418/httpatomoreillycomsourceoreillyimages1740020.png" alt="Expression parse tree" class="block-image"></div><p>Figure&nbsp;1-1.&nbsp;Expression parse tree</p></div><p>Multiplication has higher precedence than addition, so the first
343  two expressions are <code>1 * 2</code> and <code>3 * 4</code>. Then those two
344  expressions are added together, and that sum is then added to 5. Each
345  branch of the tree shows the relationship between the tokens or subtrees
346  below it. The structure of this particular tree is quite simple and
347  regular with two descendants under each node (that’s why we use a
348  calculator as the first example), but any bison parser makes a parse
349  tree as it parses its input. In some applications, it creates the tree
350  as a data structure in memory for later use. In others, the tree is just
351  implicit in the sequence of operations the parser does.</p><div><h3>BNF Grammars</h3><p>In order to write a parser, we need some way to describe
352  the rules the parser uses to turn a sequence of tokens into a parse
353  tree. The most common kind of language that computer parsers handle is
354  a <span><em>context-free grammar</em></span> (CFG).<sup>[<a href="https://www.oreilly.com/library/view/flex-bison/9780596805418/ch01.html#ftn.id302969">3</a>]</sup> The standard form to write down a CFG is
355  <span><em>Backus-Naur Form</em></span> (BNF), created around 1960 to
356  describe Algol 60 and named after two members of the Algol 60
357  committee.</p><p>Fortunately, BNF is quite simple. Here’s BNF for simple
358  arithmetic expressions enough to handle <code>1 * 2 + 3 * 4 + 5</code>:</p><div style="overflow: auto;"><pre>&lt;exp&gt; ::= &lt;factor&gt;
359  | &lt;exp&gt; + &lt;factor&gt;
360 &lt;factor&gt; ::= NUMBER
361  | &lt;factor&gt; * NUMBER</pre></div><p>Each line is a <span><em>rule</em></span> that says how to create
362  a branch of the parse tree. In BNF, <code>::=</code> can be read “is a” or “becomes,” and
363  <code>|</code> is “or,” another way to create a
364  branch of the same kind. The name on the left side of a rule is a
365  <span><em>symbol</em></span> or <span><em>term</em></span>. By
366  convention, all tokens are considered to be symbols, but there are
367  also symbols that are not tokens.</p><p>Useful BNF is invariably quite recursive, with rules that refer
368  to themselves directly or indirectly. These simple rules can match an
369  arbitrarily complex sequence of additions and multiplications by
370  applying them recursively.</p></div><div><h3>Bison’s Rule Input Language</h3><p>Bison rules are basically BNF, with the punctuation
371  simplified a little to make them easier to type. <a href="https://www.oreilly.com/library/view/flex-bison/9780596805418/ch01.html#calc1">Example&nbsp;1-5</a> shows the bison code, including the BNF, for the
372  first version of our calculator.</p><div><p>Example&nbsp;1-5.&nbsp;Simple calculator fb1-5.y</p><div style="overflow: auto;"><pre>/* simplest version of calculator */
373 %{
374 #include &lt;stdio.h&gt;
375 %}
376 
377 /* declare tokens */
378 %token NUMBER
379 %token ADD SUB MUL DIV ABS
380 %token EOL
381 
382 %%
383 
384 calclist: /* nothing */ matches at beginning of input
385  | calclist exp EOL { printf("= %d\n", $2); } EOL is end of an expression
386  ;
387 
388 exp: factor default $$ = $1
389  | exp ADD factor { $$ = $1 + $3; }

403 {
404  yyparse();
405 }
406

408 {
409  fprintf(stderr, "error: %s\n", s);
410 }</pre></div></div><p>Bison programs have (not by coincidence) the same three-part
411  structure as flex programs, with declarations, rules, and C code. The
412  declarations here include C code to be copied to the beginning of the
413  generated C parser, again enclosed in <code>%{</code> and <code>%}</code>. Following that are <code>%token</code> token declarations, telling bison the
414  names of the symbols in the parser that are tokens. By convention,
415  tokens have uppercase names, although bison doesn’t require it. Any
416  symbols not declared as tokens have to appear on the left side of at
417  least one rule in the program. (If a symbol neither is a token nor
418  appears on the left side of a rule, it’s like an unreferenced variable
419  in a C program. It doesn’t hurt anything, but it probably means the
420  programmer made a mistake.)</p><p>The second section contains the rules in simplified BNF. Bison
421  uses a single colon rather than <code>::=</code>, and since line boundaries are not
422  significant, a semicolon marks the end of a rule. Again, like flex,
423  the C action code goes in braces at the end of each rule.</p><p>Bison automatically does the parsing for you, remembering what
424  rules have been matched, so the action code maintains the values
425  associated with each symbol. Bison parsers also perform side effects
426  such as creating data structures for later use or, as in this case,
427  printing out results. The symbol on the left side of the first rule is
428  the <span><em>start symbol</em></span>, the one that the entire input
429  has to match. There can be, and usually are, other rules with the same
430  start symbol on the left.</p><p>Each symbol in a bison rule has a value; the value of the target
431  symbol (the one to the left of the colon) is called <code>$$</code> in the action code, and the values on the
432  right are numbered <code>$1</code>, <code>$2</code>, and so forth, up to the number of
433  symbols in the rule. The values of tokens are whatever was in <code>yylval</code> when the scanner returned the token;
434  the values of other symbols are set in rules in the parser. In this
435  parser, the values of the <code>factor</code>,
436  <code>term</code>, and <code>exp</code> symbols are the value of the expression
437  they represent.</p><p>In this parser, the first two rules, which define the symbol
438  <code>calclist</code>, implement a loop that
439  reads an expression terminated by a newline and prints its value. The
440  definition of <code>calclist</code> uses a
441  common two-rule recursive idiom to implement a sequence or list: the
442  first rule is empty and matches nothing; the second adds an item to
443  the list. The action in the second rule prints the value of the
444  <code>exp</code> in $2.</p><p>The rest of the rules implement the calculator. The rules with
445  operators such as <code>exp ADD factor</code>
446  and <code>ABS term</code> do the appropriate
447  arithmetic on the symbol values. The rules with a single symbol on the
448  right side are syntactic glue to put the grammar together; for
449  example, an <code>exp</code> is a <code>factor</code>. In the absence of an explicit action
450  on a rule, the parser assigns <code>$1</code> to
451  <code>$$</code>. This is a hack, albeit a very
452  useful one, since most of the time it does the right thing.</p><div><p>How About Parsing English?</p><p>For a very long time, as far back as the 1950s, people have

504  grammar in <a href="https://www.oreilly.com/library/view/flex-bison/9780596805418/ch01.html#calc1">Example&nbsp;1-5</a> is needlessly complicated. Why not
505  just write this?</p><div style="overflow: auto;"><pre>exp: exp ADD exp
506  | exp SUB exp
507  | exp MUL exp
508  | exp DIV exp
509  | ABS exp
510  | NUMBER
511  ;</pre></div><p>There are two answers: precedence and ambiguity. The separate symbols for <code>term</code>, <code>factor</code>, and <code>exp</code> tell bison to handle <code>ABS</code>, then <code>MUL</code> and <code>DIV</code>, and then <code>ADD</code> and <code>SUB</code>. In general, whenever a grammar has
512  multiple levels of precedence where one kind of operator binds “tighter”
513  than another, the parser will need a level of rule for each
514  level.</p><p>Well, then, OK, how about this?</p><div style="overflow: auto;"><pre>exp: exp ADD exp
515  | exp SUB exp
516  | factor
517  ;
518  similarly for factor and term</pre></div><p>One of bison’s greatest strengths, and simultaneously one
519  of its most annoying aspects, is that <span><em>it will not parse an
520  ambiguous grammar</em></span>. That is, any parser that bison creates has
521  exactly one way to parse any input that it parses, and the parser will
522  accept exactly that grammar. The previous grammar is ambiguous, because
523  input such as <code>1 - 2 + 3</code> could be
524  parsed either as <code>(1-2) + 3</code> or as
525  <code>1 - (2+3)</code>, two different expressions
526  with different values. Although there are some cases where ambiguity
527  doesn’t matter (e.g., <code>1+2+3</code>), in most
528  cases the ambiguity really is an error, and the grammar needs to be
529  fixed. The way we wrote the grammar in <a href="https://www.oreilly.com/library/view/flex-bison/9780596805418/ch01.html#calc1">Example&nbsp;1-5</a> makes
530  expressions unambiguously group to the left. If a grammar is ambiguous,
531  bison reports <span><em>conflicts</em></span>, places where there are two
532  different possible parses for a given bit of input. It creates a parser
533  anyway, picking one option in each conflict, but that choice means the
534  language it’s parsing isn’t necessarily the one you tried to specify. We
535  discuss this at length in <a href="https://www.oreilly.com/library/view/flex-bison/9780596805418/ch07.html">Chapter&nbsp;7</a>.</p><p>Bison’s usual parsing algorithm can look ahead one token to decide
536  what rules match the input. Some grammars aren’t ambiguous but have
537  places that require more than one token of lookahead to decide what
538  rules will match. These also cause conflicts, although it is usually
539  possible to rewrite the grammar so that one token lookahead is
540  enough.</p><p>Actually, the previous discussion about ambiguity is not quite
541  true. Since expression grammars are so common and useful, and since
542  writing separate rules for each precedence level is tedious, bison has
543  some special features that let you write an expression grammar in the
544  natural way with one rule per operator in the form <code>exp OP exp</code> and just tell it what precedence
545  and grouping rules to use to resolve the ambiguity. We’ll learn about
546  these in <a href="https://www.oreilly.com/library/view/flex-bison/9780596805418/ch03.html">Chapter&nbsp;3</a>. Also, bison has an alternative
547  parsing technique called GLR that can handle ambiguous grammars and
548  arbitrary lookahead, tracking all the possible parses that match the
549  input in parallel. We cover this in <a href="https://www.oreilly.com/library/view/flex-bison/9780596805418/ch09.html">Chapter&nbsp;9</a>.</p></div><div><h2>Adding a Few More Rules</h2><p>One of the nicest things about using flex and bison to

551  changes to the grammar. Our expression language would be a lot more
552  useful if it could handle parenthesized expressions, and it would be
553  nice if it could handle comments, using <code>//</code> syntax. To do this, we need only add one
554  rule to the parser and three to the scanner.</p><p>In the parser we define two new tokens, <code>OP</code> and <code>CP</code>
555  for open and close parentheses, and add a rule to make a parenthesized
556  expression a term:</p><div style="overflow: auto;"><pre>%token OP CP in the declaration section
557  ...
558 %%
559 term: NUMBER

561  | OP exp CP { $$ = $2; } New rule

563  parentheses, to <code>$$</code>.</p><p>The scanner has two new rules to recognize the two new tokens and

569  calculator, are both simple enough that we could without too much

572  flex uses is quite fast and is usually about the same speed as a
